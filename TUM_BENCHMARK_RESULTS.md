# TUM RGB-D Benchmark Evaluation

## Overview

This document records the results of running the RGB-D pose tracking pipeline
against two TUM RGB-D benchmark sequences to establish an objective trajectory
quality metric (Absolute Trajectory Error).  This is the foundation for the
automated parameter tuning system: every AWS Batch experiment will report the
same ATE numbers computed here.

## Sequences Tested

| Sequence | Frames (RGB/Depth) | Duration | Description |
|---|---|---|---|
| `fr1/room` | 1362 / 1360 | ~48 s | Short room-scale, moderate motion |
| `fr3/long_office_household` | 2585 / 2509 | ~87 s | Long office walk, slower motion |

## Pipeline Configuration

Both sequences use TUM-specific configs generated by `tum_to_dataset.py`:

- **DEPTH_SCALE**: 5000.0 (TUM uint16 PNGs: pixel 5000 = 1 m)
- **Resolution**: 640x480 at ODO_SCALE=0.5 (odometry at 320x240)
- **Intrinsics**: Per-sensor calibration, zero distortion (TUM recommendation:
  images and depth are pre-registered by the OpenNI driver, so undistorting
  the depth maps is not meaningful)
- **Motion thresholds**: Relaxed for handheld TUM sequences
  (MAX_VELOCITY=2.0 m/s, MAX_ANGULAR_VEL=5.0 rad/s)

## ATE Results (evo_ape tum --align --correct_scale)

Evaluation uses Umeyama Sim(3) alignment to account for the arbitrary
reference frame difference between the pipeline's output and TUM's
motion-capture ground truth.

| Metric | fr1/room | fr3/long_office |
|---|---|---|
| **RMSE** | **0.1206 m** | **0.0394 m** |
| Mean | 0.1084 m | 0.0349 m |
| Median | 0.0915 m | 0.0311 m |
| Max | 0.2328 m | 0.1162 m |
| Min | 0.0175 m | 0.0061 m |
| Std | 0.0529 m | 0.0182 m |
| Scale correction | 0.984 | 0.997 |

### Interpretation

**Scale correction near 1.0** confirms that DEPTH_SCALE=5000 is correct.
Earlier runs with the wrong depth scale (1000, the pipeline default) produced
scale corrections of 12x-30x and RMSE above 0.9 m.

**fr1/room (RMSE 0.121 m):** Reasonable for a handheld sequence with
significant rotation.  The max error of 0.23 m occurs during fast turns.
For context, state-of-the-art methods on this sequence report RMSE of
0.04-0.08 m; our pipeline is within 2-3x of that without any sequence-specific
tuning.

**fr3/long_office (RMSE 0.039 m):** Excellent.  The slower, more translational
motion of this sequence plays to the pipeline's strengths.  This result is
competitive with published benchmarks for frame-to-frame odometry methods.

### What constitutes "good" vs "poor" for room-scale?

- **< 0.05 m RMSE**: Excellent -- competitive with state of the art
- **0.05 - 0.15 m RMSE**: Good -- suitable for room-scale 3D reconstruction
- **0.15 - 0.50 m RMSE**: Marginal -- visible drift, loop closures may help
- **> 0.50 m RMSE**: Poor -- likely a configuration error (depth scale,
  intrinsics) or the tracker is diverging

## Reproducing These Results

```bash
# 1. Convert TUM sequence to pipeline format
python tum_to_dataset.py dataset/rgbd_dataset_freiburg1_room --output dataset

# 2. Generate file lists
python 1.make_file_lists.py

# 3. Run pose tracking with TUM-specific config
python -u 2.pose_tracking.py --config tum_fr1_config.json

# 4. Evaluate with evo
pip install evo
evo_ape tum dataset/groundtruth.txt dataset/pose_trajectory.txt \
    --align --correct_scale -v

# For fr3, substitute the appropriate paths and config
```

## Key Gotchas Discovered

1. **Depth scale is critical.** TUM uses 5000 (pixel 5000 = 1 m), not the
   pipeline default of 1000.  Wrong depth scale produces RMSE > 0.5 m and
   scale corrections of 10-30x.

2. **Do not undistort TUM images.** TUM's OpenNI driver pre-registers depth
   to RGB and pre-undistorts.  Applying distortion correction to already-clean
   images corrupts the depth maps and causes tracking fragmentation.

3. **Motion thresholds need relaxing for handheld.** The pipeline defaults
   (MAX_VELOCITY=0.5, MAX_ANGULAR_VEL=2.5) are tuned for a Jetson-mounted
   robot.  TUM's handheld sequences need ~2.0 / 5.0 to avoid fragmenting
   into many short segments.

4. **The config loading feature (`--config`) was not on the main branch.**
   It was added as part of this work to enable sequence-specific parameter
   overrides.

5. **fr3 has no accelerometer data.**  The pipeline falls back gracefully
   (gyro integration returns identity), so this is not a problem.
